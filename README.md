# Travel Reimbursement Approval Agent

A GenAI-powered travel reimbursement approval system that evaluates employee travel claims using a controlled hybrid agentic workflow.

The system uses **Gemini** for tool planning, policy interpretation, expense-to-policy mapping, and final reimbursement recommendation. It uses **deterministic Python tools** for auditable execution, including receipt validation, duplicate detection, eligibility checks, policy limit calculation, approved/rejected amount calculation, and approval routing.

This design keeps the LLM responsible for reasoning over natural-language policy, while Python tools handle the checks and calculations that must be exact, traceable, and safe.

## Project Overview

The **Travel Reimbursement Approval Agent** evaluates employee travel reimbursement claims against a company travel reimbursement policy.

The system accepts a submitted travel claim and returns:

- Final reimbursement decision
- Submitted amount
- Approved amount
- Rejected amount
- Approval routing level
- Expense-level decisions
- Policy references
- Final reimbursement recommendation generated by Gemini
- Tool execution evidence
- Full audit trace through API output and UI dropdowns

The project includes:

- FastAPI backend
- Swagger API documentation
- Simple browser-based UI
- Gemini-powered tool planning
- Gemini-powered policy reasoning
- Deterministic Python tools
- Command-line demo runner
- JSON and Markdown based sample data

---

## Core Idea

This project follows a **hybrid GenAI + deterministic tool architecture**.

A pure Python rule-based system is rigid because real company policies are usually written in natural language.

A pure LLM-based system is risky because reimbursement decisions involve money, compliance, receipts, duplicate claims, approval thresholds, and exact calculations.

So this project separates responsibilities clearly:

| Layer | Responsibility |
|---|---|
| Gemini | Tool planning, policy interpretation, expense-to-policy mapping, final recommendation |
| Python tools | Receipt checks, duplicate checks, eligibility checks, limit calculation, approved/rejected amount calculation, approval routing |
| Python response safety layer | Preserves Gemini explanation, injects deterministic tool results, validates final response structure |
| FastAPI | API layer and browser UI |
| Pydantic | Data validation and structured response models |

The goal is not to let the LLM blindly approve claims. The goal is to let Gemini reason over policy text while Python tools execute the financial checks safely.

---

## Agent Workflow

The reimbursement evaluation runs through the following workflow:

| Step | Component | Owner | Purpose |
|---:|---|---|---|
| 1 | Tool Planning | Gemini | Decides which reimbursement tools are required for the claim |
| 2 | Tool Plan Validation | Python | Ensures mandatory checks are not skipped |
| 3 | Policy Reasoning | Gemini | Maps each expense to applicable policy IDs |
| 4 | Tool Execution | Python tools | Performs deterministic validation and calculations |
| 5 | Final Recommendation | Gemini | Generates the final reimbursement recommendation using tool evidence |
| 6 | Response Safety Layer | Python | Injects deterministic tool-calculated values and validates the final response structure |

## Gemini Model and Quota Note

This project supports using Gemini models through the Google GenAI SDK.

The recommended model is:

```text
gemini-2.5-flash
```

A lighter alternative is:

```text
gemini-2.5-flash-lite
```

A full LLM-enabled evaluation can use up to three Gemini API calls:

```text
1. Tool planning
2. Policy reasoning
```

For final demo testing, use:

```bash
python run_demo.py
```

or open:

```text
http://127.0.0.1:8000/ui
```

## Python Response Safety Layer

The final recommendation is generated by Gemini, but the final API response is passed through a Python response safety layer.

This is mainly handled in:

```text
app/response_builder.py
app/models.py
app/agent.py
```

The purpose of this layer is to prevent unsafe or inconsistent final outputs.

Gemini is allowed to write:

- Final decision wording
- Final reimbursement explanation
- Business-friendly recommendation
- Confidence score, when valid

Python controls and validates:

- Claim ID
- Employee ID
- Employee name
- Submitted amount
- Approved amount
- Rejected amount
- Approval level
- Missing documents
- Manual review reasons
- Reason codes
- Expense-level decisions
- Tool execution evidence
- Final response structure

This means that even if Gemini writes an explanation, Python tools remain the source of truth for reimbursement amounts and approval routing.

Conceptually:

```text
Gemini = semantic reasoning and explanation
Python tools = deterministic calculation
Python response safety layer = structure and tool-truth enforcement
```

## Fallback Behavior

The system is designed to handle temporary LLM failures safely.

| Failure Case | Behavior |
|---|---|
| Gemini tool planning fails | Python uses a safe deterministic tool plan |
| Gemini policy reasoning fails | Claim is routed to Manual Review or safe fallback policy reasoning is used |
| Gemini final recommendation fails | A safe deterministic fallback response can be returned based on Python tool results |

This is useful when the Gemini API is temporarily unavailable, rate-limited, or quota-exhausted.

---

## Design Decision

This project intentionally separates reasoning from execution.

```text
Gemini interprets policy.
Python tools calculate money and validate evidence.
Gemini writes the final recommendation.
Python response safety layer keeps structured outputs valid and traceable.
```

This avoids two common problems:

1. Pure Python rules are too rigid for natural-language policies.
2. Pure LLM systems are risky for financial calculations.

The hybrid approach provides both flexibility and safety.

## Interview Explanation

A concise explanation:

```text
This is a controlled hybrid agentic reimbursement approval system. Gemini is used for tool planning, policy reasoning, and final recommendation generation. Deterministic Python tools are used for receipt validation, duplicate detection, eligibility checks, approved/rejected amount calculation, and approval routing. This allows the system to interpret natural-language policy using an LLM while keeping financial calculations auditable and safe.
```

If asked why Python tools are needed:

```text
The LLM should not be trusted for exact reimbursement calculations. Python tools ensure approved and rejected amounts are calculated deterministically and consistently.
```

If asked why Gemini is needed:

```text
Real company policies are often written in natural language. Gemini helps map each claim expense to the relevant policy rules without converting the entire policy document into hardcoded if-else logic.
```

If asked what makes the project agentic:

```text
The system uses Gemini to plan which tools are needed, then Python validates and executes those tools. Gemini then uses the tool evidence to produce the final reimbursement recommendation.
```

If asked about the response safety layer:

```text
The LLM generates the final explanation, but Python validates the response structure and injects deterministic tool-calculated values such as approved amount, rejected amount, approval level, reason codes, and expense-level decisions.
```
